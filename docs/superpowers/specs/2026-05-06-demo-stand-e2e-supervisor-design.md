# Демо-стенд CPLANE (миграция 000008): честный E2E со супервизором

## Контекст

Цель — улучшить демонстрационный стенд в миграции `000008_demo_stand_seed` так, чтобы в одном сценарии были видны: подключённые датасеты, несколько моделей в пайплайне, использование параметров датасетов в запросе к супервизору, отправка задачи через RabbitMQ, наблюдаемый прогресс и итог **без подделки статусов/исполнения в БД**.

Репозитории: **SKIF-CPLANE** (БД, backend, frontend, compose) и **skif_platform_supervisor** (Java-супервизор, каталог `development/supervisor/models`). Супервизор ожидается рядом с CPLANE для `docker-compose` (`../skif_platform_supervisor`).

## Принятые решения (из согласования)

| Вопрос | Решение |
|--------|---------|
| Реалистичность | Честный end-to-end с реальным супервизором и Docker-моделями; фейковые job/артефакты в Postgres не использовать. |
| Область изменений | CPLANE + супервизор + связанный Docker/compose при необходимости. |
| Топология | Один общий `docker compose` со **shared network** (сервисы по именам: `rabbitmq`, `backend`, `java-supervisor`). |
| Датасеты в CPLANE | Тип **JSON** (`params` в БД); отдельный «файловый тип датасета» в CPLANE для демо не требуется. |
| Подход к объёму работ | **Подход 1** (минимально достаточный E2E); без обязательного третьего узла пайплайна. |
| «Результат» в UI | Отображение **стадий и статуса** через HTTP API супервизора (`GET /api/experiments/{id}/status`). Авто-запись итога в **`t_dataset_v`** как отдельный продуктовый автоматический write-back за scope этого спека не заявлен. |

## Архитектура потока данных

1. **CPLANE backend** загружает полное описание эксперимента (`CompleteExperimentInfo`), включая связанные датасеты через **`t_experiment_dataset`**.
2. Пакет `internal/pkg/supervisor` собирает **ExperimentRequest** и вызывает **`attachDatasetsToModelParameters`**: в `parameters` каждой модели добавляются **`datasets`** и (если ключи ещё не заданы в шаблоне) массивы **`input_datasets`** / **`output_datasets`** с объектами вида `{ alias, id, params, schema, … }`, классифицированными по имени алиаса (substring `input`, `output`, …).
3. Конфиг публикуется в **RabbitMQ** под ключ эксперимента (`PublishExperimentStart`).
4. **Супервизор** принимает JSON, упорядочивает модели по `order`, для каждой строит **start.json** в **StartJsonFactory**: `HardcodedModelDefaults` ← выход предыдущего шага (`previousOutput`) ← `model.getParameters()` из CPLANE. Запускается **`AbstractDockerModelExecutor`**: биндинги `/input/start.json`, `/output`, `/json`.
5. **Цепочка из двух шагов**: первая модель пишет `/output/end.json`; супервизор требует его наличие и переименование в том числе под порядковый файл; второй шаг получает смёрдженный `previousOutput`.

## Ключевые инварианты и риски

### Шаблон эксперимента не должен блокировать обогащение датасетами

Если в JSON шаблона в таблице уже присутствуют ключи **`input_datasets`** или **`output_datasets`**, код CPLANE их **не перезаписывает**. Тогда в супервизор уходят строковые плейсхолдеры вроде `["beam_input"]` вместо полных объектов с **`params`**.

**Требование:** в сиде демо-шаблона убрать из `parameters` моделей эти ключи (или оставить только то, что осознанно нужно без CPLANE-обогащения), чтобы сработало автоматическое заполнение из БД.

### Содержательная связь «датасет → первая модель»

Текущий **`model_01` (Python)** читает из `start.json` поля **`E_input`**, **`h_y_1`**, …, которые по умолчанию задаёт **`HardcodedModelProvider`** по номеру шага. Для демонстрации **влияния датасета** нужно небольшое изменение Python-модели: например, если в JSON есть обогащённые **`input_datasets`** / **`datasets`**, взять числовой параметр (например **`energy_keV`** из `params` первого input) и отобразить его в используемое **`E_input`**, иначе — оставить текущее поведение (дефолты супервизора).

### Таблица `t_experiment_io`

В текущем SQL для `CompleteExperimentInfo` используются связи через **`t_experiment_dataset`**. Отдельное наполнение **`t_experiment_io`** не обязательно, пока нет потребителя в API; перед добавлением строк в миграцию — быстрая проверка использования в коде.

### Инфраструктура

- **`docker-compose.yml`** в CPLANE уже описывает **RabbitMQ**, **backend**, **java-supervisor**, **host workdir** для JSON и **docker.sock** для вложенных контейнеров.
- Для статуса в UI backend должен иметь **`clients.supervisor.base_url`**, достижимый из контейнера **backend** (например `http://java-supervisor:8080` в той же сети compose).

## Ошибки и наблюдаемость

- Ошибки публикации в RabbitMQ — пользователю через API (существующая логика).
- Неверный или пустой **supervisor base URL** — пользователь видит сообщение о недоступности статуса; в инструкции к демо указать проверку URL и логов `java-supervisor`.
- Сбой Docker/отсутствие `end.json` — виден в логах супервизора и в статусе эксперимента; не маскировать фиктивными данными.

## Критерии приёмки

1. После `docker compose up` и миграций в неймспейсе **`demo-stand-skif`** виден проект с экспериментом и **двумя привязанными датасетами** с понятными алиасами.
2. Запуск пайплайна из CPLANE приводит к **реальному** выполнению **двух** моделей в супервизоре (логи + отсутствие искусственных записей о завершении в БД).
3. В UI отображаются **стадии** и **итоговый статус** супервизора по **HTTP API**.
4. Изменение числового параметра во **входном** датасете в CPLANE приводит к **наблюдаемому** изменению поведения первого шага (например иной эффективный `E_input` / иной выход первой модели), проверяемому логами или косвенно по цепочке второго шага.

## Вне scope

- Третья модель в пайплайне «для красоты».
- Фиктивные строки job/статуса в Postgres.
- Полноценный write-back результатов симуляции в **`t_dataset_v`** без отдельного продуктового решения и API.
- Сценарий **B**: разрозненные сети без общего compose (достаточно зафиксировать, что демо рассчитано на **A**).
